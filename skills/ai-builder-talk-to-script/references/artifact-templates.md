# AI Builder 口播：工作模板与文件规则

## 1. 输入清单

在写作前记录：

| 输入角色 | 选择的文件或来源 | 版本性质 | 覆盖率或风险 |
| --- | --- | --- | --- |
| 英文逐字稿 |  | 清洗稿 / raw / 官方稿 |  |
| 双语逐字稿 |  | 对照稿 / 机器翻译 / 人工校正 |  |
| PPT |  | 官方 / 视频投屏复原 |  |
| 幻灯片索引 |  | `slides.json` / 无 |  |
| 视频元数据 |  |  |  |
| 讲者与活动信息 |  | 官方活动页 / 官方简介 / 其他 |  |
| 项目规范 |  |  |  |

如果 PPT 是视频投屏复原版，不得称为官方原始 PPT。文件名中出现 `_原PPT` 只表示它在项目中的材料角色，不能单独证明官方来源；继续核对 `SOURCE.md`、来源链接和页面结构。

扁平图片型 PPT 没有现成幻灯片索引时，使用 `scripts/build_slide_index.py` 生成最小 `slides.json`。该索引只记录文件页码、页数和结构检查，不虚构主题、时间戳或官方来源状态。

## 2. 证据映射表

先在临时空间建立：

| ID | 类型 | 主张、案例或工具 | 逐字稿证据 | PPT 页码 | 视频时间 | 口播用途 | 置信度 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| S01 | `S` 原分享证据 |  |  |  |  | 人物 / 故事 / 观点 |  |
| I01 | `I` Elise 解读 | 由哪些 `S` 推导 |  |  |  | Builder 判断 |  |
| T01 | `T` 本系列工具 | 由哪些 `S`、`I` 提炼 |  | 自制工具卡 | — | 方法交付 |  |

规则：

- `S` 才能写成“讲者说”或“讲者的方法”。
- `I` 使用“我理解”“我听完的判断”等清楚归属。
- `T` 使用“我把它整理成……”并说明不是讲者正式命名。
- PPT 只能证明画面内容；直接引语回到英文逐字稿核验。
- 不确定的事实保留“待核验”，不要补写成顺畅但虚假的叙事。

## 3. 口播稿文件名

信息完整时使用：

```text
【AI Builders解读NN】YYYYMMDD_Speaker_OriginalTalkTitle_解读视频逐字稿_草稿.md
```

`YYYYMMDD` 是解读视频的计划首次发布日期，不是原演讲日期。日期尚未确定时，不要猜测；保留安全工作名，等用户确认后再改。

复杂内容可增加：

```text
<相同前缀>_解读证据与画面映射_草稿.md
<相同前缀>_解读配套工具箱_草稿.md
```

草稿写入 `【YouTube学习资料】/` 下对应原分享的加工资料夹，不写入 `【AI Builders解读NN】...` 正式归档资料夹。可丢弃的渲染、预览和检查产物写入根目录 `output/`。

## 4. 口播稿骨架

```markdown
# 【AI Builders解读NN】讲者｜中文解读标题｜解读视频逐字稿草稿

> 状态：口语化工作草稿，确认后再进入正式归档。
> 预计口播时长：约 X—Y 分钟。
> 原文来源：[文件或链接]。
> 画面口径：官方 PPT / 视频投屏截图复原版。
> 证据映射：生成后填写实际相对链接。
> 配套工具：生成后填写实际相对链接。

## 开场｜具体标题

> 画面：……

讲者是谁、相关背景与分享现场。

具体冲突或问题。

固定 Elise 系列识别句。

本期可以反复使用的具体工具。

结尾会回答的问题。

## 第一幕｜具体标题

> 画面：原 PPT 第 X—Y 页；画面内容。

具体案例 → 冲突 → 原观点 → Builder 解读 → 工具 → 转场问题。

## 第二幕｜具体标题

> 画面：……

……

## 收尾｜回收开场问题

> 画面：……

回答开场问题。

一句 Builder 判断。

提醒可保存的工具与具体互动问题。
```

没有证据映射或工具箱时，删除对应链接，不保留空字段。

## 5. 工具箱骨架

```markdown
# 【AI Builders解读NN】配套工具箱｜草稿

> 来源：根据【讲者】在【活动】的分享【标题】提炼。
> 说明：以下工具由本期 AI Builders 解读整理，不是讲者在原分享中使用的正式名称。

## 工具一｜工具名称

### 来源关系

### 什么时候使用

### 输入

### 使用步骤

### 输出

### 使用边界
```

工具有多项时，逐项重复来源关系、适用场景、输入、步骤、输出和边界。长提示词放在代码块中，可直接复制；口播只讲用途、关键输入和输出。

## 6. 交付前核对

### 来源与归属

- [ ] 人物履历、职位、活动信息已核验。
- [ ] 每个数字、原话和产品细节可回查。
- [ ] 引号内文字与英文原文一致。
- [ ] Elise 的问题和工具没有写成讲者原话或原方法。
- [ ] 投屏复原 PPT 已明确说明不是官方原始 PPT。

### 开头与结构

- [ ] 前一分钟能回答：谁、在哪里分享、为什么值得听、能带走什么。
- [ ] 钩子具体、无需前情、正文有证据、结尾有回收。
- [ ] 固定系列识别句位于人物和现场之后、具体收益之前。
- [ ] 正文由故事推进，不是逐页摘要或概念清单。

### 方法交付

- [ ] 开头承诺、正文、工具箱与结尾的工具名称、数量、顺序一致。
- [ ] 每项工具包含场景、输入、步骤、输出、边界和来源关系。
- [ ] 工具帮助作出判断，而不只是布置一次性动作。
- [ ] AI 的辅助范围与 Builder 的最终责任写清楚。

### 听感与时长

- [ ] 抽象概念先有案例，再被命名并用白话解释。
- [ ] 每个案例无需专业术语前置也能理解；必要术语已先用白话说明作用。
- [ ] 隐藏标题和画面提示后，正文仍能第一次听懂。
- [ ] 无翻译腔、长复句、僵硬排比或重复总结。
- [ ] 字数和预计时长匹配用户要求。

### 文件边界

- [ ] 全部工作文件带 `_草稿` 并留在加工资料夹。
- [ ] 文件之间的相对链接有效。
- [ ] 未经用户确认没有写入或覆盖正式归档。
